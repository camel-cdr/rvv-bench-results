RVV benchmark MangoPi MQ Pro (Allwinner D1, C906)

The MangoPi MQ Pro has a single Allwinner D1 XuanTie C906 core, which has been open-sourced. It has a frequency 1GHz and implements rv64gc and the 0.7.1 version of the vector extension supporting SEW=8/16/32, single and half precision floating point instructions. The VLEN is 128 bits, but the ALU is 256 bits wide.

Performance observations

Note that all of these only apply to this processor and aren't set in stone.

  • Maximizing LMUL is a good strategy in general, but isn't always the fastest, see mergelines.html. This is likely due to vcompress.vm not scaling proportionally to LMUL (4, 10, 36, 136 cycles).
  • The vl doesn't impact performance, so a vl=1 and a vl=VLMAX instruction with LMUL=8 will take the same amount of time to execute.
  • Using the mask operand doesn't impact performance.
  • Most instructions take 4 for LMUL=1/2, 8 for LMUL=4 and 16 for LMUL=8 cycles to execute.
  • The fault-only-first load instructions are quite slow compared to the regular load instructions, so one should try to page align the loads by hand when possible. (see strlen)
  • Manual tail handling and unrolling can increase the performance for certain sizes (~20% in utf8 count), when the operation isn't memory bound.
  • Aligning the destination pointer in a streaming workload, like memcpy can make the runtime slightly faster/more stable.

Problems and unexpected things

  • There seems to be a problem with some kernel builds, reducing the clocks peed considerably.
  • Even though the C906 chip is supposed to support RVV 0.7.1, it dosn't implement the vdot/vfdot instructions.
  • vmsgt.vx v8, v8, a3 gives an illegal instruction, even though it is supposed to work. This probably also includes other instructions, but vmsgt.vx with different source and result operands seems to work.
  • mstatus.vs is masked by 0x01800000 instead of 0x600, as specified for RVV 1.0. (RVV 0.7.1 didn't specify a position)
