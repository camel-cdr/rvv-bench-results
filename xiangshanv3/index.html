RVV benchmark XiangShanV3 (Kuminghu)

XiangShanV3 also called Kuminghu, is the third generation of the OpenXiangShan open-source RISC-V processor project. It's a high-performance super-scalar out-of-order core that targets performance comparable to the ARM Neoverse N2 processors, and a 45@3GHz SPECInt2006 score. See the latest tutorials page on the documentation website for slides including more architectural details. Also be sure to check out the source code; hint Parameters.scalar gives a good high-level overview of the core.
Kuminghu introduced RVV support to XiangShan, it is still work in progress, and further optimizations and fixes are expected.

The current Kuminghu RVV implementation is severely bottlenecked by instruction decode. The master branch can only decode one vector instruction per cycle, while the scalar part of the core can decode and issue six instructions per cycle. Since the vector backend has five execution units, you basically have to write LMUL>=4 code to take advantage of the full backend.
However, mask-generating LMUL>1 instructions (e.g. comparison instructions) currently generate dependent uops, which causes catastrophic LMUL latency scaling. Other currently not well implemented instructions include: vslideup/vslidedown, vcpop.m, vid.v and segmented-load/stores.
The lmul-1 branch adds support for decoding simple LMUL=1 vector instructions at the full decode width of six. This notably doesn't include .vf instruction variants and any other instruction that generates more than one uop at LMUL=1. vsetvli instructions are still only decoded at one per cycle, which limits the performance of not manually unrolled LMUL=1 loops.

All measurements were done on rtl simulation of the lmul-1 branch from 2025-05-08.
